# P0 UX 개선 설계

> 작성일 2026-06-23 · 상태: 설계 승인됨, 플랜 대기
> 맥락: 조각 A/C/D 완료 후 사용자 관점 UX 심층 진단에서 도출한 P0(핵심 마찰) 묶음.
> 비주얼은 **현재 프레임 유지**(라이트 테마, `--primary:#3366ff`, Pretendard, 둥근 카드). [[wanted-matcher-ui-prefers-existing-frame]]

## 문제 / Go

웹 UI는 동작하지만 일상 사용에 다음 마찰이 있다:
1. 이력서(약 6,000자)를 세션마다 재입력 — persistence 없음.
2. 채점은 공고당 ~10–30초(gemma 12b) → 기본 50개면 10–25분인데 **ETA·중단 수단 없음**.
3. 0건/에러 시 침묵하거나 날것 메시지("Unexpected token '<'...") → "고장났나?" 오해(실제 발생).

**안 만들면:** 재입력·깜깜이 대기·혼란이 그대로. 도구를 본인+친구가 반복 사용하므로 마찰 제거 가치 큼 → 진행(Go).

| 항목 | 결정 |

## ① 이력서 + 검색조건 저장/복원 (프론트 전용)

- `localStorage` 키: `wm.resume`, `wm.query`, `wm.tagIds`(JSON 배열), `wm.limit`.
- **저장 시점:** 이력서는 `input` 이벤트 디바운스(예: 400ms) 저장. query/tags/limit는 `run()` 시작 시 저장(현재 검색을 다음 방문에 복원).
- **복원 시점:** 페이지 로드 시 이력서·query·limit 즉시 채움. 칩은 `/tags` 렌더 완료 후 저장된 `tagIds`와 대조해 `selectedTags`에 넣고 `.selected` 클래스 부여.
- 백엔드 변경 없음. 민감정보지만 개인 로컬 도구라 수용(이력서는 이미 브라우저에 평문 입력).

## ② 서버측 진짜 중단 (프론트 + 백엔드)

**프론트(`app.js`/`index.html`):**
- 실행 중 "분석 시작" 버튼이 "중단"으로 전환(또는 옆에 중단 버튼 노출). 클릭 시 진행 중 `fetch`의 `AbortController.abort()` 호출 → SSE 연결 종료.
- 중단 시 상태줄 "중단됨" 표시, 이미 받은 카드는 유지. 잠금이 풀리므로 즉시 재검색 가능.

**백엔드:**
- `src/server.ts` `/run`: 핸들러에서 `const ac = new AbortController()`. 정상 완료를 표시하는 `finished` 플래그 도입. `req.on('close', () => { if (!finished) ac.abort(); })`. `ac.signal`을 `runPipeline`에 전달.
- `src/pipeline.ts` `runPipeline(resume, limit, onProgress, deps?, excludeIds?, search?, signal?)` — `signal`을 `fetchJobs`와 `scoreJobs`로 전달. `PipelineDeps` 시그니처 확장.
- `src/wanted.ts` `fetchJobsWithDetails(limit, excludeIds, search, signal?)` — 상세 fetch 루프 각 반복 시작에서 `if (signal?.aborted) break;`.
- `src/scorer.ts` `scoreAllJobs(jobs, resume, onScored?, scoreOne?, signal?)` — 배치 루프 각 반복 시작에서 `if (signal?.aborted) break;` (이미 채점된 결과만 반환).
- 중단 시 `runPipeline`은 부분 결과를 반환하고 정상 복귀 → 서버 `finally`가 `running=false` 해제. 클라이언트는 이미 연결을 끊었으므로 잔여 이벤트 전송 실패는 무시.

## ③ 빈 / 0건 결과 상태 (프론트 전용)

- 결과 영역(`#results`)에 상태 메시지 컴포넌트 1개(`#emptyState`).
- **첫 진입(아직 실행 안 함):** "이력서를 넣고 직군·검색어를 고른 뒤 분석을 시작하세요."
- **0건 완료:** "조건에 맞는 신입 공고를 찾지 못했어요." + 실행 가능한 팁 목록: 키워드를 줄이거나 빼기 / 다른 직군 칩 선택 / 공고 수 늘리기.
- 카드가 1개 이상이면 숨김. "완료 — 0개 공고" 텍스트만 두던 침묵을 대체.

## ④ 에러 메시지 친화화 (프론트 전용)

- `friendlyError(raw)` 매퍼 추가. 매핑:
  - 응답 본문이 JSON이 아님 / `Unexpected token '<'` / HTML 응답 → "서버 응답이 올바르지 않습니다. 서버를 재시작했는지 확인하세요(옛 서버가 떠 있을 수 있습니다)."
  - `Failed to fetch` / 네트워크 → "서버에 연결할 수 없습니다. 서버가 실행 중인지 확인하세요."
  - 그 외/서버 error 이벤트(예: Ollama 다운 메시지) → 원문 노출(이미 한국어 친화적).
- `/tags` 로딩 실패: 페이지 로드 즉시 빨간 배너 대신 **칩 영역 인라인 메시지 + "재시도" 버튼**. (P3-15 동시 해결)

## ⑤ 기본 공고 수 15 + 예상시간 / ETA (프론트 전용)

- `index.html` limit 기본값 `50` → `15`.
- limit 입력 옆 힌트: "약 N분 예상"(N = `ceil(limit * 15초 / 60)`, 입력 변경 시 갱신). 상수 15초는 대략치(주석 명시).
- 실행 중 상태줄 ETA: 첫 `scored` 이후 경과시간 기반 `(elapsed/done)*(total-done)`로 "채점 중 d/total · 약 N분 남음". 0건 수집 시엔 ETA 생략.

## ⑥ 비주얼 일관화 (프론트 전용, 현재 프레임 내)

- 새 요소(중단 버튼·`#emptyState`·ETA 힌트·인라인 에러·재시도)는 기존 클래스 톤(rem, `var(--primary)`, 둥근 모서리, `--border`/`--muted`)에 맞춤.
- 뒤늦게 추가돼 어긋난 스타일 정리: `.query-input`/`.chip`/`.chip.selected`/`.reset-btn`의 하드코딩 색(`#2f6fed`, `#d1d5db`, `#374151` 등)·px를 CSS 변수·rem으로 통일. **레이아웃/프레임은 그대로.**

## 테스트

- `scorer.scoreAllJobs`: 이미 abort된 signal → 채점 0회. 진행 중 abort(2개째 전 abort) → 처리된 것만 반환.
- `wanted.fetchJobsWithDetails`: abort 시 상세 fetch 루프 조기 종료(주입형 fetch mock으로 호출 횟수 검증).
- `pipeline.runPipeline`: `signal`을 `fetchJobs`·`scoreJobs`로 전달.
- `server`: SSE 스트림 중 클라이언트 연결 종료(요청 abort) → 잠금 해제되어 다음 `/run`이 200.
- 프론트(persistence/empty/error/ETA/힌트): 자동 테스트 없음 → 브라우저 수동 검증(체크리스트는 플랜 마지막 태스크).

## RAID

| 구분 | 항목 |
|---|---|
| Risk | `req.on('close')`는 정상 완료에도 발화 → `finished` 플래그로 정상 완료 시 abort 방지. |
| Risk | 중단 후 부분 결과 반환 시 `done` 전송이 실패할 수 있음(연결 끊김) → write 실패 무시, 잠금 해제만 보장. |
| Assumption | `AbortController`/`AbortSignal`은 현 Node 런타임 기본 제공(18+). tsx 환경 OK. |
| Assumption | gemma 채점 평균 ~15초/공고는 대략치(ETA 힌트용). 실제 ETA는 경과시간 기반이라 자동 보정됨. |
| Dependency | 기존 SSE/seen/searchSpec 동작 불변. CLI는 `signal` 미전달(기본 undefined)로 불변. |
| Out of scope | 결과 localStorage 복원(P1-7), 결과 필터/정렬(P1-6), 요약 통계(P1-5), 접근성 전면(P3-17). |
