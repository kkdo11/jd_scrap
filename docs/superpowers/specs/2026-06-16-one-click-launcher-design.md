# Wanted Job Matcher — 원클릭 런처 설계

**상태:** 승인됨 (구현 대기)
**작성일:** 2026-06-16
**목표:** 현재 다단계 수동 실행(Ollama 켜기 → 모델 확인 → resume.txt 준비 → `npm start` → 브라우저로 report.html 열기)을, **Windows 바탕화면 `.bat` 더블클릭 한 번**으로 안정성 점검·자동 복구까지 포함해 끝내는 로컬 원클릭 런처를 만든다.

> 참고: `2026-05-26-daily-driver-design.md`의 GitHub Actions 매일 자동화는 별도의 큰 비전이며, 이 문서의 범위가 **아니다.** 이 문서는 본인 PC에서 즉시 돌리는 로컬 원클릭 경험에 집중한다.

---

## 1. 환경 전제

- 개발/실행 환경: **WSL2 (Windows 위의 Linux)**, 셸 `bash`
- 리포 위치: WSL 파일시스템 (`~/scrap`)
- "한 번 클릭" = **Windows 바탕화면의 `.bat` 더블클릭** (WSL 사용자에게 가장 '진짜 원클릭'에 가까움)
- 점검 정책: **최대한 자동 복구** (단, 시스템 레벨 설치는 안내만)

---

## 2. 선택된 접근: Option B — 얇은 `.bat` + 진짜 로직은 `run.sh`

브레인스토밍에서 2안 비교:

- ~~A. 단일 `.bat`에 모든 로직~~ — 배치 문법 빈약, WSL/리눅스 점검 로직 중복·취약, 디버깅 어려움.
- ✅ **B. 얇은 `.bat` 래퍼 + `run.sh` 메인** — 로직이 한 곳(bash)에 모이고, 터미널에서 `./run.sh`로도 동일 실행 가능, 점검을 네이티브 bash로 견고하게.

**선택 이유:** 로직 단일화 + 견고함 + 이식성. `.bat`은 단지 WSL 안으로 들어가 `run.sh`를 호출하는 진입점일 뿐.

---

## 3. 구성 (산출물)

| 파일 | 역할 |
|---|---|
| `run.sh` | 메인 런처. 모든 안정성 점검 + 자동 복구 + 실행 + 리포트 열기. 실행 권한 부여(`chmod +x`). |
| `win/원티드매칭.bat` | Windows 더블클릭용 얇은 래퍼. 사용자가 바탕화면에 복사하거나 바로가기 생성. |
| `run.log` | 실행 로그(자동 생성). `.gitignore`에 추가. |
| `README.md` (갱신) | "원클릭 실행" 섹션 추가 — `.bat`을 바탕화면에 두는 법, 동작 설명. |

### `win/원티드매칭.bat` (개략)

```bat
@echo off
chcp 65001 > nul
wsl -e bash -lc "cd ~/scrap && ./run.sh"
echo.
pause
```

- `chcp 65001` = 한글 출력 깨짐 방지(UTF-8)
- `pause` = 끝나거나 에러로 멈춰도 창이 사라지지 않게(로그·메시지 확인 가능)
- 경로 `~/scrap`는 현 리포 위치 기준. README에 "다른 경로면 이 줄을 고치세요" 안내.

## 4. `run.sh` 실행 흐름

각 단계는 화면에 한국어 `✓ / ✗ / …` 상태를 출력하고 동시에 `run.log`에 기록한다.

| # | 점검 | 문제 시 동작 |
|---|---|---|
| 1 | `node` / `npm` 존재 | ✗ 없으면 **중단** + 설치 안내. (시스템 설치는 자동 안 함) |
| 2 | `node_modules/` 존재 | 없으면 `npm install` 자동 실행 |
| 3 | `resume.txt` 존재 & 비어있지 않음 | 없으면 **템플릿 생성** 후 **중단** + "내용을 채우고 다시 실행하세요" 안내 |
| 4 | `ollama` 명령 존재(설치됨) | ✗ 없으면 **중단** + 설치 명령 안내. (`curl \| sh` 자동 실행은 보안상 안 함) |
| 5 | Ollama 서버 응답(`http://localhost:11434`) | 꺼져있으면 `ollama serve`를 백그라운드로 시작 + 준비될 때까지 폴링(타임아웃 있음) |
| 6 | 모델(`${OLLAMA_MODEL:-exaone3.5:7.8b}`) 존재 | 없으면 `ollama pull` 자동 (~5GB, 진행률 그대로 표시) |
| 7 | 본 실행 `npm start` (인자 있으면 전달) | 실패 시 `run.log` 경로 안내하고 종료 |
| 8 | `report.html` 열기 | Windows 기본 브라우저로 자동 오픈 (`explorer.exe "$(wslpath -w report.html)"`) |

### 단계 상세 메모

- **2 (npm install):** `node_modules` 디렉터리 존재 여부로 판단(간단·충분). 설치 실패 시 중단.
- **3 (resume.txt):** 존재하지만 공백/거의 빈 경우도 "비어있음"으로 간주(바이트 수 임계값). 템플릿에는 학력/경력/스택/프로젝트 헤더만 넣어 사용자가 채우기 쉽게.
- **5 (ollama serve):** 먼저 `curl -s localhost:11434`로 살아있는지 확인. 죽어있을 때만 `nohup ollama serve >> run.log 2>&1 &`로 띄우고, **우리가 띄운 PID만 기억**. 준비 폴링은 최대 N초(예: 30s) 후 실패 처리.
- **6 (model pull):** `ollama list`에 모델명이 있는지 grep. 없으면 pull. pull 진행률은 ollama 자체 출력을 그대로 보여줌(멈춘 것처럼 안 보이게).
- **8 (open report):** `report.html`이 실제로 생성되었는지 확인 후 오픈. `wslpath -w`로 Windows 경로 변환 → `explorer.exe`로 기본 브라우저 오픈.

## 5. 안정성 설계

- `set -euo pipefail` + `trap`으로 **어느 단계에서 죽든** 마지막에 친절한 한국어 에러 메시지 + `run.log` 경로를 보여주고 종료. `.bat`의 `pause` 덕분에 창이 즉시 닫히지 않음.
- **데몬 비침습:** 사용자가 이미 켜 둔 Ollama 서버는 건드리지 않음. 우리가 직접 띄운 경우에만 추적(필요 시 정리 정책은 "그대로 둠" — 다음 실행 재사용이 더 편하므로 강제 종료 안 함).
- **멱등성:** 여러 번 더블클릭해도 안전. 이미 충족된 단계(설치됨/켜짐/모델 있음)는 건너뜀.
- **로깅:** 모든 출력이 화면 + `run.log`에 동시 기록되어, 문제 발생 시 로그로 원인 추적.
- **타임아웃:** 서버 준비 폴링 등 무한 대기 가능한 지점에 타임아웃을 둠.

---

## 6. 편의성 설계

- 더블클릭 한 번 → 끝나면 Windows 기본 브라우저에 `report.html` 자동 표시.
- 각 단계 진행상황을 한국어 ✓ 메시지로 실시간 표시(특히 `npm install`·모델 pull 같은 긴 작업에서 "멈춘 듯" 보이지 않게).
- 기존 환경변수(`OLLAMA_MODEL`, `OLLAMA_HOST`, `SCORER_CONCURRENCY`, `USE_CACHE`)와 CLI 인자(공고 수)를 그대로 존중 — 기존 사용법을 깨지 않음.
- 터미널 사용자는 `./run.sh`로 동일하게 실행 가능(이중 경로).

---

## 7. 명시적으로 안 하는 것 (이번 범위 제외)

- Node.js / Ollama **자체의 시스템 설치 자동화** (권한·보안 이슈 → 안내만).
- GitHub Actions 매일 자동 실행 / 푸시 알림 (별도 비전, `2026-05-26-daily-driver-design.md` 트랙).
- GUI 앱 형태(Electron 등)나 웹 서버화 — 단순 런처에 과함(YAGNI).
- 공고 수를 묻는 대화형 프롬프트 — 기본값 유지, 필요 시 인자로 전달.

---

## 8. 테스트 / 검증 관점

- **정상 경로:** 모든 전제 충족 상태에서 더블클릭 → 끝까지 실행 → 리포트 오픈.
- **자동 복구 경로:** `node_modules` 삭제 / Ollama 종료 / 모델 미존재 각각에서 자동 복구 확인.
- **실패 경로:** `resume.txt` 없음 → 템플릿 생성 + 친절한 중단. `node` 없음 → 안내 후 중단. 서버 폴링 타임아웃 → 명확한 에러.
- **멱등성:** 연속 두 번 실행해도 안전.
- 단, 본 리포에 자동화 테스트 인프라는 없으므로(README상 `tsc --noEmit`만), 이번엔 수동 검증 체크리스트로 충분.
